# Bug Report: Power-On Initialization (POI) Registers Bug in Vivado 2025.x

## Table of Contents

---

## Overview

This repository documents a bug in Vivado's handling of **Power-On Initialization (POI)** for registers generated by Vitis HLS. The issue causes constant register values to silently default to zero in hardware, despite being defined as non-zero constants in the original HLS code. The failure is silent, with no obvious warnings or errors. Furthermore, the registers are present (i.e. they are not optimized away) in the routed netlist with valid drivers. 

This bug was first observed in **Vitis HLS/Vivado 2025.1**. I have also observed the issue in Vivado 2025.2, but would need to resynthesise the design and rerun the test to confirm. Designs prior to Vivado/Vitis HLS 2025.1 work reliably as expected - we have tested our design extensively on the following Vitis/Vivado versions: 2022.1, 2022.2, 2023.1, 2023.2, 2024.1, 2024.2.

Importantantly, the issue seems to be in Vivado (and NOT in Vitis HLS): I've tested the same design by compiling the HLS code with Vitis HLS 2024.1 and running synthesis with Vivado 2025.1, observing the same issue.

**N.B.:** Given the last point, it is possible that the issue applies to non-HLS POI registers; however, I never use POI registers when writing RTL (wouldn't even know how to do it); hence I have not observed it in pure RTL; only when using HLS.

A similar issue has been independently reported by another Coyote user: [fpgasystems/Coyote#202](https://github.com/fpgasystems/Coyote/issues/202); though in this case, it was with Vivado 2022.1 (and 2022.1 for us works reliably).

We found a workaround for this bug (see below); however, it is possible that other projects may face similar silent failures that may be non-trivial to debug.

## Example problem: `arp_server_subnet` module

### Module Description

The module `arp_server_subnet_top_generate_arp_pkg_512_s`, generated through HLS compilation, generates ARP packets on a 512-bit wide AXI-Stream bus. The TDATA of the AXI Stream is set from a 336-bit header register (`header_header_1`) with:

- **Dynamic fields**: sender/target MAC addresses and IP addresses — written by the FSM at runtime.
- **Constant fields**: standard ARP/Ethernet protocol identifiers — assigned once in the HLS C++ constructor and never modified thereafter --> **THESE FIELDS ARE NOT CORRECTLY SET**

---

### HLS Source

The HLS source is given in: `hw/services/network/hls/arp_server_subnet`

**N.B.:** Make sure to clone the repository with submodules, as this the HLS network stack is part of a seperate repository: https://github.com/fpgasystems/fpga-network-stack (Coyote uses the coyote-TCP-RDMA branch which already includes the proposed fix)

The `arpHeader` constructor sets the following constants, which map to bits **158 to 96** of the output word `arpDataOut`:

```cpp
arpHeader() {
    header(111, 96)  = 0x0608;  // Ethernet type (ARP)
    header(127, 112) = 0x0100;  // Hardware type (Ethernet)
    header(143, 128) = 0x0008;  // Protocol type (IPv4)
    header(151, 144) = 6;       // Hardware address length
    header(158, 152) = 4;       // Protocol address length
}
```

---

### Hardware Observation — ILA Capture
The ARP server module is instantiated in `hw/hdl/network/stack/network_stack.sv` as `arp_server_subnet_ip arp_server_inst`. The `arpDataOut` variable corresponds to the `m_axis` interface, per the port definition in the HLS code.

Probing `arpDataOut[158:96]` with an ILA built with Vivado 2025.x, we get:

| Field | Bits | Expected (achieved in 24.1) | Observed in 25.1|
|---|---|---|---|
| Ethernet type | 111:96 | `0x0608` | `0x0000` |
| Hardware type | 127:112 | `0x0100` | `0x0000` |
| Protocol type | 143:128 | `0x0008` | `0x0000` |
| HW addr length | 151:144 | `0x06` | `0x00` |
| Protocol length | 158:152 | `0x04` | `0x00` |

The full 512-bit TDATA of the ARP AXI Stream is in Vivado 2024.1 is:
```
512'h564afd0aaa3ec6ebc008544afd0ab8230b350a0002000406000801000608b8230b350a00aa3ec6ebc008
```

and for Vivado 2025.1 is:
```
512'h14afd0aaa0000002020544afd0ab8230b350a0002000000000000000000b8230b350a00aa0000002020
```

Dynamic fields (MAC addresses, IP addresses) are populated correctly, confirming the FSM is functioning and only the POI-dependent constants are affected. ILA captures (CSV and native ILA capture) for both working (2024.x) and regressed (2025.x) builds are provided in `ila-data/<vivado-version>/` for direct comparison.

**N.B.:** The dynamic fields are different in the two values - the ping test was conducted from different source nodes, hence the IP, MAC addresses differ. This is normal; however, the constant values for Vivado 2025.1 are effectively set to zero.

---

### Checkpoint Inspection

The post-PnR checkpoints confirm that the registers driving bits [158:96] **are physically present in the design and have not been optimized away**. For example, we can analyze bits [99:96], with expected value 4'h8 = 4'b1000:
    - Bits 96-98: expected 0, removed from checkpoint (as expected)
    - Bit 99: expected 1, still present in checkpoint (as expected)

The checkpoints are found in `checkpoints/<vivado-version>/`

### Generated RTL Analysis — Vitis HLS 2024.1

> **File**: `verilog/2024.1/arp_server_subnet_top_generate_arp_pkg_512_s.v`

The `initial` block initialises `header_header_1` to zero. The protocol constants are embedded as a hardcoded 63-bit literal in a combinational assignment, making them independent of `header_header_1`'s initial value:

```verilog
initial begin
    #0 header_header_1 = 336'd0;
end

assign tmp_i_fu_269_p5 = {{header_header_1[335:159]}, {63'd289919260388492808}, {header_header_1[95:0]}};
```

The literal `63'd289919260388492808` (`0x0406_0008_0100_0608`) encodes all five fields:

| Bits (within literal) | Value | Field |
|---|---|---|
| [15:0] | `0x0608` | Ethernet type (ARP) |
| [31:16] | `0x0100` | Hardware type (Ethernet) |
| [47:32] | `0x0008` | Protocol type (IPv4) |
| [55:48] | `0x06` | Hardware address length |
| [62:56] | `0x04` | Protocol address length |

**First packet** — `guard_variable` starts at 0 (POI), so `sendWord_data` selects `tmp_i_fu_269_p5`:

```
arpDataOut[158:96]
  ← empty_reg_509[158:96]              (registered in iter0, output in iter1)
  ← empty_fu_460_p3[158:96]
  ← ap_phi_mux_sendWord_data[158:96]   (guard = 0, selects tmp_i_fu_269_p5)
  ← tmp_i_fu_269_p5[158:96]
  = 63'd289919260388492808             ✓  hardcoded literal, correct
```

**Subsequent packets** — `guard_variable` is 1, so `sendWord_data` selects `header_header_1` directly. The constants survive into `header_header_1` via the write path. Tracing the reply case:

```
header_header_1[158:96]  (written in iter0 from previous cycle)
  ← tmp_6_i_fu_399_p5[158:96]
  ← tmp_5_i2_fu_365_p5[158:96]        (bits [158:96] fall within the [335:96] section, above myMacAddress at [95:48])
  ← tmp_4_i1_fu_353_p5[158:96]        (bits [158:96] above trunc_ln98 boundary at [47:0])
  ← sendWord_data[158:96]             (tmp_4_i takes sendWord_data[335:48])
  = 63'd289919260388492808             ✓  constants preserved through write
```

The request path (`tmp_10_i_fu_330_p5` → `tmp_9_i4_fu_304_p5` → `tmp_8_i3_fu_292_p5`) follows the same structure: `tmp_8_i3[335:48] = sendWord_data[335:48]`, so `header_header_1[158:96]` is identically preserved.

The 2024.1 Verilog seems functionally correct.

### RTL Analysis — Vitis HLS 2025.1

> **File**: `verilog/2025.1/arp_server_subnet_top_generate_arp_pkg_512_s.v`

The `initial` block and the constant literal are identical to 2024.1:

```verilog
initial begin
    #0 header_header_1 = 336'd0;
end

assign tmp_i_fu_313_p5 = {{header_header_1[335:159]}, {63'd289919260388492808}, {header_header_1[95:0]}};
```

**First packet** — `guard_variable` starts at 0, captured into `_reg_496` in `iter0`. In `iter1`, `_reg_496_pp0_iter0_reg = 0` selects `tmp_i_fu_313_p5`:

```
arpDataOut[158:96]
  ← zext_ln147_1_cast_fu_457_p3[158:96]   (combinational output)
  ← empty_fu_445_p3[158:96]
  ← ap_phi_mux_sendWord_data[158:96]       (guard_reg = 0, selects tmp_i_fu_313_p5)
  ← tmp_i_fu_313_p5[158:96]
  = 63'd289919260388492808                 ✓  hardcoded literal, correct
```

**Subsequent packets** — `guard_variable` is 1, captured into `_reg_496` in `iter0`, so `sendWord_data` selects `header_header_1` directly. Tracing the reply case through the 2025.1 write path:

```
header_header_1[158:96]  (written in iter1 from previous cycle)
  ← tmp_6_i_fu_396_p5[158:96]
  ← tmp_5_i_fu_378_p4[158:96]             (bits [158:96] within [335:96] section, above myMacAddress at [95:48])
  ← tmp_4_i_fu_369_p4[158:96]             (bits [158:96] above trunc_ln98_reg boundary at [47:0])
  ← sendWord_data[158:96]                 (tmp_4_i takes sendWord_data[335:48])
  = 63'd289919260388492808                 ✓  constants preserved through write
```

The request path (`tmp_10_i_fu_356_p5` → `tmp_9_i_fu_336_p4` → `tmp_8_i_fu_326_p4`) is structurally identical: `tmp_8_i[335:48] = sendWord_data[335:48]`, so `header_header_1[158:96]` is preserved in the same way.

The 2025.1 Verilog also seems functionally correct

## Current Workaround

The issue can be bypassed by adding the following to the HLS project configuration (in `make.tcl.in`):

```tcl
config_rtl -reset all -reset_level low
```

This connects the constant registers to the ap_rst signal, driving them to their correct values via hardware logic on reset assertion rather than relying on the POI value.

While this workaround works fine, it is not ideal for several reasons:

- Identifying it is as a solution was non-trivial, and can be time-consuming for other projects that aren't aware of the subtle failure.

- It can increase resource consumption/timing closure complexity, by having to connect more registers to the  reset signal. In our case, this wasn't a big issue, as our network stack is relatively small and targets large, datacenter FPGAs (e.g., U55C)

## Reproducing the Bug
This repository contains everything needed to observe the bug directly: pre-built bitstreams, probe files, routed checkpoints etc. In case you want to resynthesise the bitstream and rerun the example, please follow the steps below.

### Building the Hardware and Compiling the Driver

The bug can be reproduced by building [Example 9 (`perf_rdma`)](https://github.com/fpgasystems/Coyote/tree/master/examples). The steps below are a brief summary — refer to the [Coyote examples README](https://github.com/fpgasystems/Coyote/tree/master/examples) for full prerequisites and detailed instructions.


From the Coyote root, configure and build the hardware for Example 9:

```bash
cd examples/09_perf_rdma/hw && mkdir build && cd build
cmake .. -DFDEV_NAME=u55c   # u250, u280 should also work
make project && make compile
```

Once synthesis and implementation complete, the bitstream will be in `build/bitstreams/cyt_top.bit`. To compile the Coyote driver:

```bash
cd driver
make
```

### Loading the Bitstream and Running the Test

> **Note:** The FPGA must be physically connected to a network switch or peer host via its QSFP port before proceeding. The [AMD-ETH HACC cluster](https://systems.ethz.ch/research/data-processing-on-modern-hardware/hacc.html) is a well-suited environment for this test, as the Alveo cards are pre-wired to a high-speed network with assigned IP addresses.

**1. Program the FPGA** using Vivado Hardware Manager (GUI or script)

**2. Rescan PCIe devices** to re-enumerate the PCIe device and correctly assign BARs to the FPGA. Alternatively, a **warm reboot** can also be done (BUT NOT cold boot).

**3. Insert the driver.** Because Example 9 uses FPGA networking, the IP and MAC addresses of the QSFP interface must be passed at load time:

```bash
sudo insmod Coyote/driver/build/coyote_driver.ko ip_addr=$qsfp_ip mac_addr=$qsfp_mac
```

**4. Verify ARP is working** by pinging the FPGA's assigned IP from another host on the same network:

```bash
ping <fpga-ip>
```

A working build (Vivado 2024.x) will respond correctly. A regressed build (Vivado 2025.x) will not — ARP replies are malformed because the protocol constant fields are zero, preventing the FPGA from correctly identifying itself on the network.
